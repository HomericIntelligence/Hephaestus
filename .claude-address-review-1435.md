#1435 [refactor] Add StatusTracker.slot() context manager for worker slots

## Motivation

The `status_tracker.acquire_slot()` → `status_tracker.update_slot(slot_id, msg)` → `status_tracker.release_slot(slot_id)` pattern is repeated 100+ times across 10 modules. Every worker function follows the same lifecycle:

```python
slot_id = self.status_tracker.acquire_slot()
try:
    self.status_tracker.update_slot(slot_id, "Starting")
    # ... do work ...
    self.status_tracker.update_slot(slot_id, "Done")
finally:
    self.status_tracker.release_slot(slot_id)
```

This is a classic resource management pattern that should be a context manager. If any code path forgets `release_slot()`, the slot leaks and the worker pool shrinks.

## Proposed Solution

Add a context manager to `StatusTracker`:
```python
@contextmanager
def slot(self, initial_msg: str = "") -> Iterator[int]:
    slot_id = self.acquire_slot()
    try:
        if initial_msg:
            self.update_slot(slot_id, initial_msg)
        yield slot_id
    finally:
        self.release_slot(slot_id)
```

Then every call site becomes:
```python
with self.status_tracker.slot("Starting") as slot_id:
    # ... do work ...
    self.status_tracker.update_slot(slot_id, "Done")
```

This eliminates 100+ `try/finally` blocks and prevents slot leaks.

## Impact

- Eliminates 100+ `try/finally` blocks across 10 files
- Prevents slot leaks from forgotten `release_slot()` calls
- Cleaner resource management
- Low risk — context manager wraps existing API

## Files affected

- `hephaestus/automation/status_tracker.py` (add context manager)
- All 10 modules with `acquire_slot`/`release_slot` patterns
